01

Introduction

AquaMesh is a cloud-native environmental monitoring platform built to centralize sensor data and give organizations real-time operational visibility across distributed monitoring networks. From a single web interface, teams can monitor environmental conditions, manage deployments, configure automated alerts, review historical trends, and generate AI-assisted operational insights.

The platform runs on a serverless cloud architecture, chosen specifically to support scalable deployments while keeping infrastructure management to a minimum. That same architecture adapts across a wide range of environmental monitoring applications, from coastal water quality to watershed management, aquaculture, and industrial environmental monitoring.

Monitoring is the foundation, not the ceiling. As AquaMesh's recommendations prove themselves against a site's own data, the platform can take on progressively more of the routine response itself — moving operators from constantly interpreting readings to managing exceptions on a system that's already handling the routine.

Environmental & Public Sector Monitoring

AquaMesh is deployed today across coastal and marine water quality monitoring, river and watershed monitoring, flood and drought risk assessment, reservoir and lake monitoring, aquaculture operations, and research and environmental data collection.

Industrial Water Operations

On the industrial side, AquaMesh is built around a narrower problem: every gauge on a plant floor is reading, but nobody owns the decision. The platform learns a site's own water fingerprint, catches costly deviations early, and — as trust builds — takes over the routine response, cutting water-related operating cost typically by 10 to 25 percent.

Platform Architecture

AquaMesh is built on a cloud-native, serverless architecture. Environmental observations are ingested securely through a managed API layer, where backend processing services validate incoming data, evaluate alert conditions, generate AI-assisted summaries, and coordinate platform workflows. Operational and historical data live in a managed cloud database, while a dedicated identity service handles user authentication and access management.

Separating the frontend, API layer, backend services, and data storage into independently managed layers lets the platform scale automatically with workload, without added infrastructure overhead. This is what allows organizations to manage multiple monitoring deployments through one centralized dashboard, while keeping every tenant's data securely isolated.

Integration & Interoperability

AquaMesh is built to sit above the systems a site already has, not replace them. The platform connects to existing SCADA and control systems, ingests data from third-party sensors and telemetry, and layers in AquaMesh's own AquaSpectra sensing wherever additional visibility is needed. External connections pass through the same authenticated API layer used by the frontend, so every integration inherits the same access controls and tenant isolation as the rest of the platform.

Core Platform Components

Frontend

The AquaMesh web application is the operator's centralized interface for monitoring conditions and managing deployments. Built with React and TypeScript, it delivers real-time dashboards, interactive maps, historical visualizations, operational reports, AI-generated summaries, and alert management through a responsive browser-based experience.

  • Fleet overview dashboards
  • Deployment management and interactive monitoring maps
  • Historical data visualization and report generation
  • Alert management and AI-assisted operational insights

Backend

The backend runs entirely on serverless cloud infrastructure. A managed API layer handles all client communication, while backend processing services execute the business logic behind data processing, alert evaluation, AI workflows, and report generation. Operational data is stored in a managed cloud database, and a dedicated identity service manages user authentication. Each service scales independently, which keeps the platform highly available as usage grows.

  • Processing incoming environmental observations and managing deployments
  • Evaluating alert conditions and generating operational reports
  • Supporting AI-assisted workflows
  • Enforcing user authentication and authorization

Data Management

Environmental information is organized in a hierarchical data model that groups monitoring assets by organization, deployment, hub, and sensor. Observations, deployment metadata, alert configurations, reports, and operational records all live in a managed cloud database, structured for fast retrieval of both real-time and historical data.

AI Services

AquaMesh's AI capabilities are built to reduce the effort of interpreting environmental data, not add another dashboard to watch. Involvement can range from Monitor (data and summaries only) to Advise (specific recommended responses) to Supervise (proposed actions executed within operator-set guardrails) to full AI Autopilot (proven responses run automatically, with operators managing exceptions). At every stage, every action stays logged, bounded, and reversible — autonomy is earned one validated response at a time, never switched on all at once.

  • Environmental summaries and fleet-level operational insights
  • Automated report generation
  • Natural language operational assistance
  • Progressive autonomy, from advisory recommendations to closed-loop response
04

Platform Features

Multi-Tenant Management

AquaMesh supports multiple organizations on a single platform while keeping each customer environment logically separate. Organizations manage independent monitoring deployments with no operational data exposed across tenants.

Environmental Monitoring

The platform continuously monitors measurements from distributed stations, including water temperature, turbidity, dissolved oxygen, salinity, chlorophyll, nitrate, pH, streamflow, and water level. Every measurement appears in real-time dashboards alongside historical data, so operators can assess conditions in context, not in isolation.

Interactive Monitoring Dashboard

The dashboard gives operators centralized visibility into conditions across every monitoring location. From one unified interface, teams navigate between deployments, review historical measurements, compare parameters, and check overall fleet status.

Alert Management

Configurable alert rules let AquaMesh automatically flag conditions that need operator attention. Rules are evaluated the moment new observations are processed, so operators are notified as soon as a configured threshold is crossed, not after the fact.

  • Threshold monitoring and incident generation
  • Alert acknowledgement and resolution
  • Historical incident tracking

Reporting

The reporting module turns raw observations into structured operational documents, consolidating measurements, historical trends, alert activity, and AI-generated summaries into reports built for routine monitoring and stakeholder communication.

Predictive Monitoring

For supported monitoring scenarios, AquaMesh applies predictive models that estimate future environmental conditions from recent observational data. Forecasts sit alongside current measurements, giving operators a clearer read on what's ahead and more room to act proactively rather than reactively.

Data Quality & Validation

AquaMesh uses a layered data integrity approach covering secure cloud storage, automated QA/QC checks, device health monitoring, and operational review. Data is stored on managed cloud infrastructure rather than on local, AquaMesh-owned servers, which provides scalable storage, access control, encryption options, redundancy, and high availability.

For AI-assisted analysis, AquaMesh can use commercial-grade LLM infrastructure such as Claude, from Anthropic. Where stricter data governance is required, AquaMesh can also support more restricted deployment architectures, including customer-controlled cloud environments or self-hosted options — these can reduce AI model performance or increase deployment complexity, but they're available where needed. Full compliance detail for both our cloud infrastructure and Anthropic's infrastructure is covered in Section 10, Security.

Error Detection

AquaMesh uses two levels of error detection. The first is rules and algorithms that check raw sensor data for parameter-level issues — out-of-range values, impossible readings, flatlined signals, sudden spikes, missing data, abnormal drift, fouling indicators, communication gaps, and device health warnings.

The second is AI-assisted operational review, which looks at the bigger picture across parameters, sites, time, and context. A single dissolved oxygen value may be technically valid, for example, but if it conflicts with temperature, turbidity, fluorescence, recent trends, or neighboring sites, the system can flag it as suspicious.

Flagged Data & Confidence Scoring

When data is flagged, AquaMesh can mark the reading as low confidence, remove or down-weight it from trend analysis, notify the customer, and recommend follow-up action — sensor inspection, recalibration, cleaning, or validation sampling.

Documentation

AquaMesh can provide documentation covering the data QA/QC framework used in the platform, including validation checks, confidence scoring logic, sensor health checks, alert definitions, calibration and maintenance data handling, and how flagged data is treated in analytics. To scope the right documentation, we typically clarify whether a customer needs:

  • Internal AquaMesh QA/QC methodology
  • Data handling and cloud security architecture
  • Sensor calibration and maintenance procedures
  • Regulatory reporting support
  • AI model validation and auditability

Trend & Anomaly Detection

AquaMesh identifies concerning trends through a layered detection workflow:

  1. Data quality verification — incoming data is checked for missing values, flatlines, drift, fouling, communication issues, and physically unrealistic readings.
  2. Site-specific baseline learning — the system learns normal behavior for each monitoring location rather than relying only on generic thresholds.
  3. Trend and anomaly detection — identifying steady declines, sudden shifts, abnormal rates of change, persistent deviations, and unusual relationships between parameters.
  4. Multi-parameter analysis — comparing related signals such as dissolved oxygen, temperature, pH, conductivity, turbidity, UV absorbance, and fluorescence.
  5. Operational AI review — LLM-assisted analysis summarizes what is happening, why it may matter, what confidence level the system has, and what action may be appropriate.

Example: A Declining Dissolved Oxygen Trend

If dissolved oxygen began declining steadily, AquaMesh could issue several alert types depending on configuration: an early trend warning, an approaching-threshold warning, a critical low DO alert, an abnormal rate-of-change alert, a low-confidence sensor check alert, and an AI-generated operational summary. Alerts can include the site, parameter, current value, rate of change, severity, confidence score, related signals, likely cause, and recommended action.

"Dissolved oxygen at Site 3 has declined 18 percent over the past 6 hours and is approaching the configured warning threshold. Temperature is elevated and turbidity is also increasing. Confidence is high. Recommended action: review upstream conditions and consider field verification."

Adjustable Sensitivity

Detection sensitivity can be adjusted by site, parameter, season, storm condition, operational cycle, and risk tolerance for false alarms. Third-party sensor accuracy and calibration can also affect sensitivity. High-risk sites can use tighter detection settings to catch problems earlier; noisier environments can require stronger evidence before triggering alerts. Seasonal profiles — summer low flow, winter storm runoff, dry weather, wet weather, or known production cycles — can be configured as well. AquaMesh isn't limited to static thresholds: it combines fixed limits, site baselines, rate-of-change detection, anomaly detection, sensor confidence, and AI-assisted interpretation.

07

Data Integration & Reliability

Formats & Frequency

AquaMesh is designed to work with a range of data formats and communication methods, ingesting structured data through APIs, CSV, JSON, database exports, cloud integrations, or direct device streams depending on the customer's existing systems. Data frequency can range from near real-time to periodic uploads — critical operational monitoring typically uses frequent readings, while lower-risk monitoring can use slower intervals to reduce power, bandwidth, and storage demands.

Transmission Gaps & Missing Data

AquaMesh separates data-quality issues from communication issues. When transmission gaps occur, the system checks device health, signal strength, timestamp continuity, and communication status to classify the problem. Missing or low-confidence data can be flagged, excluded from trend analysis, or down-weighted so it doesn't create false conclusions, and persistent or operationally important gaps trigger a customer notification. Where supported by the hardware configuration, local buffering can also preserve data during temporary communication outages and sync once connectivity returns.

Storage Continuity

AquaMesh's storage layer runs on AWS. AWS provides formal service level agreements for its production services, and Amazon S3 Standard — used within AquaMesh's storage architecture — is designed for 99.999999999% data durability and 99.99% object availability over a given year. Overall platform availability depends on the full deployment architecture — application services, databases, network connectivity, device uptime, and backup configuration — and can be strengthened through redundant storage, backups, access controls, and monitoring tailored to a customer's requirements.

Operational Considerations

Scaling Across Multiple Sites

AquaMesh is designed for multi-site monitoring from one dashboard. Across a network of monitoring locations, the platform maintains site-specific baselines, device health status, alerts, data confidence scores, and trend history for each site, while also supporting cross-site comparison. This lets operators see whether a change is isolated to one location or occurring across multiple sites — distinguishing a likely local sensor issue from a site-specific water quality event or a broader regional trend. Each location can carry its own thresholds, seasonal settings, maintenance history, alert contacts, and reporting rules.

Uptime & Redundancy

Cloud reliability depends on the infrastructure and architecture selected for a given deployment. Our infrastructure providers offer formal service level agreements for production workloads, with storage designed for high durability and availability. For a production deployment, AquaMesh can configure redundancy, backup policies, monitoring, and alerting to match a customer's uptime requirements. Field-level uptime also depends on power, connectivity, site access, fouling, and maintenance conditions.

Data Governance on the Customer Side

To get the most out of the platform, customers should maintain consistent, clean records for the following:

  • Site names
  • Device IDs
  • Sensor serial numbers
  • Calibration records
  • Maintenance logs
  • Parameter names and units
  • Sampling locations
  • Time zones and timestamps
  • Alert contacts
  • Validation samples or lab results, when available

AquaMesh works with each customer to standardize these fields during onboarding. Clean metadata improves trend detection, reduces false alarms, and makes reports easier to trust.

Predictive Models & Risk Assessment

AquaMesh supports predictive risk scoring for harmful algal blooms, fish kill conditions, and marine toxicity risk, combining live sensor data, site history, weather, temperature, dissolved oxygen, turbidity, chlorophyll, fluorescence, UV absorbance, and other available inputs into AquaMesh's AI intelligence model. This capability is positioned as risk detection and decision support, not a guaranteed toxicity diagnosis — the more site history, lab results, and event records a customer can provide, the stronger model performance becomes.

Data Sources

Models can draw on both region-specific and broader datasets: AquaMesh site data, customer historical data, open-source regional water quality data, lab results, NOAA HAB records, NOAA regional forecast archives, and USGS/EPA Water Quality Portal data. Model performance is strongest when a customer can provide a large historical dataset for their own site.

Modeling Approach

AquaMesh uses a layered modeling approach: data quality validation, time-series trend detection, anomaly detection, multi-parameter risk scoring, site-specific baseline modeling, predictive models where enough historical data exists, and LLM-assisted operational interpretation. Depending on the data available, predictive models may include regression models, tree-based models, ensemble models, classification models, time-series forecasting, and site-specific risk scoring. The broader HAB research field supports the use of machine learning, remote sensing, historical bloom records, and environmental variables for detection and forecasting — one published study, for example, used historical HAB records and remote sensing data to detect and short-term forecast Karenia brevis blooms in Florida. AquaMesh's LLM layer is used for operational interpretation, reporting, and recommended action guidance — it is not the sole model deciding whether a water quality risk exists.

Historical Data & Authentication

AquaMesh sources data from its proprietary dataset, customer records, and established public systems such as NOAA, USGS, EPA, and state or local monitoring programs. For public data, AquaMesh prioritizes government and research-grade sources: NOAA HABSOS provides current and historical HAB event information, including Karenia brevis cell counts and environmental data, and the Water Quality Portal provides access to publicly available water quality data from USGS, EPA, and hundreds of partner agencies. Customer-supplied data is incorporated after review of metadata, sampling methods, lab methods, timestamps, units, location accuracy, and QA/QC records.

Configurable Thresholds

Predictive thresholds can be tuned by site, season, parameter, species or toxicity scenario, risk tolerance, and alert level — for example, summer low dissolved oxygen risk, red tide conditions, stormwater runoff events, seasonal bloom windows, fish kill risk, high-risk public health locations, or noisy and highly variable sites. The goal throughout is balancing early detection against false-alarm control: high-risk sites can use more sensitive thresholds, while noisier sites can require stronger evidence before an alert fires.

Deployment

AquaMesh runs entirely on managed cloud infrastructure. The frontend, API layer, backend services, and data storage each run as independently managed, auto-scaling services. This serverless model simplifies updates, reduces operational overhead, and scales automatically as usage grows.

13

Operational Workflow

From sensor reading to operator decision, AquaMesh follows a consistent, straightforward workflow:

  1. Environmental sensors collect measurements from monitoring locations.
  2. Sensor observations are transmitted to the AquaMesh backend.
  3. The API layer receives incoming data requests.
  4. Backend processing services validate and process incoming observations.
  5. Environmental data is stored within the platform's managed database.
  6. Alert rules are evaluated against newly received observations.
  7. The frontend retrieves updated information through the platform's API layer.
  8. AI services generate operational summaries and reports.
  9. Operators review dashboards, alerts, and reports to support decision-making.
